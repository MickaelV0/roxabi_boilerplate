## Executive Summary

Documentation-as-code has become the industry standard for developer-focused projects in 2025-2026, with 84% of developers relying on documentation but only 58% of organizations maintaining it. For the Roxabi Boilerplate (Bun/TurboRepo/TypeScript monorepo), the recommended approach combines lightweight Markdown files in the repository for core docs, MADR templates for architectural decisions, and TypeDoc for API reference generation. This maintains simplicity while enabling automation and scalability.

**Update 2026-02-03**: Added documentation server research (Fumadocs + Docker) - see section "Documentation Server Research" below.

## State of the Art (2025-2026)

### Key Trends

1. **Docs-as-Code Maturity**: Writing documentation with the same tools as code (Git, plain text, CI/CD) is now standard practice at leading tech companies including Google, Microsoft, and GitHub.

2. **AI-Powered Documentation**: AI chatbots using RAG (Retrieval Augmented Generation) deliver instant answers from documentation. Tools like Mintlify integrate AI assistants directly.

3. **Engineering Intent Focus**: Documentation increasingly captures "why" decisions were made, not just "what" was built. Missing intent is identified as the biggest bottleneck for enterprise engineering.

4. **Automated Workflows**: CI/CD integration for documentation testing, validation, and deployment prevents "documentation drift."

5. **ADRs Adoption**: Architecture Decision Records are now recommended by AWS, Microsoft Azure, and UK Government for tracking technical decisions with context and consequences.

### Best Practices

| Practice | Description |
|----------|-------------|
| Version Control | Store docs in Git alongside code |
| Plain Text Format | Use Markdown/AsciiDoc for human-readable, trackable content |
| Automated Testing | Test code examples in CI/CD |
| Single Source of Truth | One location for all documentation |
| Continuous Documentation | Auto-generate API docs from code |

## Three Options

### Option A: Minimal MVP (Markdown Only)

**Description**: Keep everything in Markdown files within the `docs/` folder. Manual maintenance, no generators, no build process.

**Structure**:
```
docs/
├── INDEX.md              # Main index (exists)
├── vision.md             # Exists
├── architecture.md       # Exists
├── configuration.md      # Exists
├── hooks.md              # Exists
├── getting-started.md    # Create (exists as draft)
├── contributing.md       # Create
├── decisions/            # ADR folder
│   ├── 000-template.md   # MADR minimal template
│   └── 001-tech-stack.md # First ADR example
└── artifacts/specs/                # Feature specifications
    └── template.md       # Spec template
```

**Implementation**:
- Create `contributing.md` with PR guidelines
- Create `decisions/` with MADR minimal template
- Create `artifacts/specs/` with basic spec template
- No build process needed

**Pros**:
- Zero dependencies
- Immediate implementation (< 1 day)
- Easy to maintain
- Works with any editor
- GitHub renders Markdown natively

**Cons**:
- No search functionality
- No API documentation automation
- Manual maintenance burden
- No versioning support
- Basic navigation only

**Complexity**: 1/5

---

### Option B: Gold Standard (VitePress + TypeDoc + Full Automation)

**Description**: Full documentation site with VitePress, automated API docs via TypeDoc, ADR tooling, and CI/CD workflows.

**Structure**:
```
docs/
├── .vitepress/           # VitePress configuration
│   ├── config.ts
│   └── theme/
├── guide/
│   ├── index.md
│   ├── getting-started.md
│   └── contributing.md
├── architecture/
│   ├── index.md
│   └── decisions/        # ADRs with MADR full template
├── api/                  # Auto-generated from TypeDoc
│   ├── web/
│   └── api/
├── artifacts/specs/
│   └── template.md
└── public/               # Static assets
```

**Implementation**:
- Install VitePress as devDependency
- Configure TypeDoc with typedoc-plugin-markdown
- Set up GitHub Actions for:
  - Build docs on PR
  - Deploy to GitHub Pages on merge
  - Auto-generate API docs
- Install adr-tools for ADR management
- Create comprehensive templates

**Pros**:
- Professional documentation site
- Full-text search
- API documentation automation
- Version management
- Dark mode, responsive design
- Great developer experience

**Cons**:
- Significant setup time (3-5 days)
- Additional dependencies to maintain
- Build process complexity
- May be overkill for early stage
- VitePress uses Vue (team uses React)

**Complexity**: 5/5

---

### Option C: Balanced Approach (Markdown + TypeDoc + Basic Automation)

**Description**: Markdown-first documentation with TypeDoc for API reference and lightweight GitHub Actions automation. No documentation site generator.

**Structure**:
```
docs/
├── INDEX.md              # Main index
├── vision.md             # Exists
├── architecture.md       # Exists
├── configuration.md      # Exists
├── hooks.md              # Exists
├── getting-started.md    # Quick start
├── contributing.md       # Contribution guide
├── api/                  # TypeDoc generated (gitignored)
│   └── README.md         # How to generate locally
├── decisions/            # ADRs
│   ├── README.md         # ADR process guide
│   ├── 000-template.md   # MADR template
│   └── 001-tech-stack.md # Initial decisions
└── artifacts/specs/                # Feature specifications
    ├── README.md         # How to write specs
    └── template.md       # Spec template
```

**Implementation**:
1. Create manual Markdown files (Day 1)
2. Set up TypeDoc with typedoc-plugin-markdown (Day 1-2)
3. Add npm scripts: `bun run docs:api` (Day 2)
4. Create GitHub Action to validate docs on PR (Day 2)
5. Add MADR template with examples (Day 2)

**GitHub Action** (`.github/workflows/docs.yml`):
```yaml
name: Documentation
on:
  pull_request:
    paths:
      - 'docs/**'
      - 'packages/*/src/**'
      - 'apps/*/src/**'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v1
      - run: bun install
      - run: bun run docs:api
      - run: bun run docs:lint  # Optional: markdownlint
```

**Pros**:
- Low complexity, high value
- API docs automated
- ADRs formalized
- CI validation prevents drift
- Stays in Markdown (GitHub native)
- Can upgrade to VitePress later
- 2-3 days implementation

**Cons**:
- No search (GitHub search works)
- No documentation site
- Less polished presentation
- Manual navigation

**Complexity**: 2/5

## Recommendation

**Recommended: Option C (Balanced Approach)**

### Justification

1. **Aligned with Project Phase**: Roxabi Boilerplate is in early MVP phase. Full documentation infrastructure (Option B) is premature. Minimal docs (Option A) won't scale.

2. **Docs-as-Code Principle**: Option C follows the vision document's stated approach: "Docs-as-code (Markdown in repo)" while adding automation for API docs.

3. **TypeScript First**: TypeDoc integrates naturally with the strict TypeScript codebase, auto-generating API reference from TSDoc comments.

4. **ADR Best Practice**: MADR templates capture architectural decisions (AWS and Microsoft recommended practice) without heavy tooling.

5. **CI/CD Ready**: GitHub Actions validation prevents documentation drift, enforcing the "documentation maintained like code" principle.

6. **Upgrade Path**: If needed later, VitePress can be added on top of existing Markdown files without restructuring.

7. **No Framework Lock-in**: Unlike Docusaurus (React) or VitePress (Vue), pure Markdown remains portable.

### Implementation Priority

1. **Immediate** (Day 1):
   - Create `contributing.md`
   - Create `decisions/` with MADR template
   - Create `artifacts/specs/` with template

2. **Short-term** (Day 2-3):
   - Set up TypeDoc configuration
   - Add `bun run docs:api` script
   - Create GitHub Action for validation

3. **Future** (Phase 2+):
   - Consider VitePress if public documentation site needed
   - Add markdownlint for style consistency
   - Consider Mintlify if AI-powered docs become valuable

## Appendix: MADR Minimal Template

Recommended template for `docs/decisions/000-template.md`:

```markdown
# [ADR-NNNN] [Short Title]

**Status**: [Proposed | Accepted | Deprecated | Superseded]
**Date**: YYYY-MM-DD
**Deciders**: [List of people involved]

## Context

[What is the issue that we're seeing that is motivating this decision?]

## Decision

[What is the change that we're proposing/making?]

## Consequences

### Positive
- [Benefit 1]
- [Benefit 2]

### Negative
- [Drawback 1]
- [Drawback 2]

### Neutral
- [Side effect that is neither positive nor negative]

## Options Considered

1. **[Option 1]**: [Brief description]
2. **[Option 2]**: [Brief description]
3. **[Option 3]**: [Brief description]

## Links

- [Related ADR or document]
- [External reference]
```

## Documentation Server Research (2026-02-03)

### Objective

Find the best solution to view Markdown documentation in a browser locally, with future upgrade path to public SaaS documentation.

### Solutions Analyzed (7 parallel research agents)

| Solution | Category | Verdict |
|----------|----------|---------|
| **Fumadocs** | React/Next.js | ✅ **Selected** - TanStack Start support |
| **Starlight** | Astro | ⚠️ Doesn't work in monorepo subdirectory |
| **VitePress** | Vue/Vite | ⚠️ Vue-focused, Bun partial support |
| **Docusaurus** | React/Webpack | ⚠️ Bundle size issues, no Vite |
| **Rspress** | React/Rust | ✅ Alternative - Best performance |
| **Docsify** | Runtime | ❌ Maintenance inactive, poor SEO |
| **Mintlify** | SaaS | ❌ $300/month, no self-host |
| **MkDocs** | Python | ⚠️ Not TypeScript-native |
| **Nextra** | Next.js | ⚠️ Less flexible than Fumadocs |
| **GitBook** | SaaS | ⚠️ Limited free tier |

### Detailed Comparison

#### Fumadocs (Selected)
- **Version**: 16.4.7 (Feb 2026)
- **Stack Match**: ⭐⭐⭐⭐⭐ - Official TanStack Start support
- **TypeScript**: First-class (77.6% codebase)
- **Bundle**: 161KB (smallest)
- **Users**: Vercel, Unkey, Orama
- **Features**: OpenAPI generator, React Server Components, i18n, versioning

#### Starlight (Alternative)
- **Version**: 0.37.6 (Feb 2026)
- **Adoption**: Cloudflare, Google, Microsoft, OpenAI
- **Performance**: &lt;50KB, zero JS, 95+ Lighthouse
- **Issue**: Does NOT work in non-root directory (`apps/docs/` impossible)
- **Note**: Cloudflare acquired Astro (Jan 2026)

#### Rspress (Alternative)
- **Backer**: ByteDance
- **Performance**: 5-10x faster builds (Rust), 20x faster Markdown parsing
- **Good for**: Maximum performance requirements

#### VitePress
- **Downloads**: 357k/week
- **Issue**: Vue-focused (project uses React), Bun runtime issues
- **Versioning**: No native support (plugin required)

#### Docusaurus
- **Stars**: 63k GitHub
- **Backer**: Meta
- **Issues**: Webpack only (no Vite), bundle size scales with routes

### Decision: Fumadocs + Docker

**Why Fumadocs**:
1. Only framework with official TanStack Start support
2. OpenAPI docs generator (useful for NestJS API)
3. Smallest bundle size (161KB)
4. First-class TypeScript
5. Active maintenance

**Why Docker**:
1. Isolates Fumadocs dependencies from monorepo
2. No pollution of `apps/` with Next.js/React versions
3. Portable and reproducible
4. Simple command: `bun docs`

### Architecture Decision

**Before (proposed initially)**:
```
apps/
├── web/
├── api/
└── docs/     ← Fumadocs as TurboRepo app
```

**After (final decision)**:
```
apps/
├── web/
├── api/
└── packages/ ← Moved from root
docs/
├── *.md      ← Content stays here
├── Dockerfile
└── docker-compose.yml
```

**Rationale**:
- `docs/` is self-contained (content + server)
- Docker isolates Fumadocs runtime
- No TurboRepo integration needed
- Simple: `docker compose up` or `bun docs`
