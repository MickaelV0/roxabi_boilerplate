## Context

This analysis details two "On the Horizon" opportunities identified from **808 Claude Code sessions** (680h compute, 624 commits, 16 days). Both proposals aim to **remove the human from the middle of chains already being run** — turning multi-session workflows into single-prompt pipelines.

| Opportunity | What it automates | You still do |
|---|---|---|
| **PR Review + Fix Pipeline** | Review → triage → fix → test → commit | Approve the final result |
| **Knowledge + Scaffolding** | Issue → branch → worktree → scaffold → PR | Review the draft PR |

## 1. Autonomous Multi-Agent PR Review and Fix Pipeline

### What You Already Do

You dispatch multi-agent code reviews on PRs, triage findings interactively, then launch parallel agents to fix them. Standout example: **19 fixes across 4 parallel agents touching 30 files** in a single session.

### Current Workflow (Manual Orchestration)

```
You → "review PR" → Claude reviews → You triage findings → You dispatch fix agents → You verify
```

### Proposed Workflow (Single Prompt)

```
You → "review & fix PR #123" → Done
```

### How It Works

The idea is a **single prompt** that chains the full pipeline:

1. **Agent 1 (Coordinator)** reads every changed file, produces a categorized issue list (bugs, style, architecture, missing tests) with severity ratings
2. **Spawns parallel fix agents** — one per category, each working on the same branch
3. Each fix agent **runs typecheck + tests after its changes**, reverts if broken
4. Coordinator **runs the full pre-push hook suite** at the end
5. Produces a summary + conventional commit — no questions asked

### Why It Matters

The #1 friction point is Claude taking wrong approaches (27 instances). With parallel agents validating against each other + automated test gates, bad approaches get caught and reverted automatically instead of you noticing 10 minutes later.

### What's Needed Today

Nothing new — the workflow already exists. It's about packaging it into a **single `/review-fix` skill** that orchestrates the full chain. The Task tool sub-agents provide the execution substrate.

### Architecture

### Copyable Prompt

```
Review PR #[NUMBER] using this workflow:

1) First, read every changed file and produce a categorized list of issues
   (bugs, style, architecture, missing tests) with severity ratings.

2) For each category, create a parallel sub-agent task that fixes all issues
   in that category on the current branch. Each sub-agent must: run the
   TypeScript typecheck and any existing tests after its changes, and revert
   if tests fail.

3) After all sub-agents complete, run the full pre-push hook suite.

4) Produce a summary of all changes made, any issues that couldn't be
   auto-fixed, and commit with a conventional commit message grouping all
   fixes.

Do not ask me questions — make reasonable decisions and document your
assumptions in the commit body.
```

### Risk Assessment

| Risk | Mitigation |
|------|------------|
| Conflicting fixes between parallel agents | Each agent works on a distinct category; coordinator resolves conflicts |
| Silent regressions | Mandatory typecheck + test after each agent's changes |
| Over-fixing (changing correct code) | Severity ratings filter out low-confidence findings |
| Wrong commit on main | Always works on the PR branch, never main |

## 2. Autonomous Knowledge Ingestion and Project Scaffolding

### What You Already Do (Separately)

- **Save links** to 2ndBrain via Telegram (20 sessions)
- **Manage GitHub issues** + project boards (8 sessions)
- **Write specs** from interviews, then scaffold features
- **Maintain `.claude/skills/`** and propagate patterns across repos

### Current Workflow (Multi-Session)

```
Save links → Create issue → Write spec → Create branch/worktree → Implement → PR
```

### Proposed Workflow (Single Prompt)

```
You → "Bootstrap feature: [description]" → Draft PR ready for review
```

### How It Works

A **bootstrap agent** that:

1. **Reads project conventions** — CLAUDE.md, skills, standards, existing patterns
2. **Searches the codebase** for the most similar existing feature (as a reference pattern)
3. **Creates a GitHub issue** with a full spec body and acceptance criteria via `gh issue create`
4. **Creates branch + worktree** following naming convention (`feat/XXX-slug`)
5. **Implements the feature** following patterns found in step 2 — types, API routes, UI components using the shared UI package
6. **Runs typecheck + build** to verify
7. **Commits, pushes, creates a draft PR** via `gh pr create --draft`
8. **Prints a summary** of everything created

### Why It Matters

Multiple sessions are currently spent going from idea to shipped PR. This collapses the **repetitive scaffolding steps** (issue creation, branch setup, boilerplate code, PR creation) into one shot. You still review the draft PR — but you skip the mechanical setup entirely.

### Key Ingredient

The existing codebase patterns. Claude **searches for similar features and replicates the structure** rather than guessing from scratch. This is what makes the scaffolding accurate — it's pattern-matching against your own code, not generating from a generic template.

### Copyable Prompt

```
Bootstrap a new feature: [FEATURE DESCRIPTION].

Follow this complete autonomous workflow:

1) Read all files in .claude/skills/ and CLAUDE.md to understand project
   conventions, tech stack, and patterns.

2) Search the existing codebase with Grep and Glob to find the most similar
   existing feature for reference patterns.

3) Create a GitHub issue with the feature spec using 'gh issue create' with
   a detailed body including acceptance criteria.

4) Create a new branch named feature/[FEATURE-SLUG] and a git worktree if
   we're in a worktree-based workflow.

5) Implement the feature following the patterns found in step 2, including:
   TypeScript types, API routes, UI components using the shared UI package,
   and any necessary database schema changes.

6) Run the full typecheck and build to verify everything compiles.

7) Commit with a conventional commit referencing the issue number, push,
   and create a draft PR with 'gh pr create --draft'.

8) Print a summary of everything created. Use the existing component
   standards doc and API client patterns already in the repo.
```

### Risk Assessment

| Risk | Mitigation |
|------|------------|
| Scaffolded code doesn't match conventions | Step 2 searches for reference pattern first; step 1 reads CLAUDE.md |
| Issue created with wrong spec | Draft PR — you review before merging |
| Wrong branch naming | Skill encodes the `feat/XXX-slug` convention |
| Build failures | Step 6 runs typecheck + build; fails before creating PR |
| Unwanted push | Uses `--draft` flag; you review before publishing |

## TL;DR

Both proposals are about **removing yourself from the middle of chains you already run**:

- **`/review-fix`**: You stop triaging PR findings manually — agents review, fix, validate, and commit autonomously
- **`/bootstrap`**: You stop setting up issues/branches/boilerplate manually — one prompt goes from idea to draft PR

The workflows already exist. The skills just **package and automate the orchestration**.
